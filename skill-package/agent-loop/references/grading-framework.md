# Grading Framework

## Philosophy

Grading is not a formality — it's the mechanism that turns first-draft output into partner-ready work. The entire value of the loop depends on honest, specific grading. A soft pass on a weak criterion is worse than useless because it creates false confidence.

The grading framework has one job: answer the question "would this survive scrutiny from the smartest, most demanding person in the room?"

## Generating Criteria

### The Anti-Generic Rule

Every criterion must be specific to THIS deliverable. If a criterion could apply to any output on any topic, it's too generic and needs to be rewritten.

**Bad criteria** (too generic):
- "The analysis is thorough"
- "Recommendations are actionable"
- "The output is well-organized"
- "Sources are cited"

**Good criteria** (specific to the task):
- "The competitive landscape section names at least 5 direct competitors with revenue estimates and recent strategic moves for each"
- "Each recommendation includes a specific dollar impact estimate or range, a timeline, and the responsible team/person"
- "The EBITDA bridge shows the walk from reported EBITDA to adjusted EBITDA with each add-back individually identified and sourced to a specific page in the CIM"
- "Every factual claim about market size, growth rates, or competitive positioning cites a specific source (analyst report, company filing, or news article) with date"

### Criterion Categories

Generate criteria across these dimensions:

**Accuracy (2-3 criteria)**
Tests whether facts, figures, and claims are correct.
- Are numbers internally consistent (do subtotals add up)?
- Are external claims verifiable?
- Are there logical contradictions?

**Completeness (2-3 criteria)**
Tests whether the output addresses every element from the execution plan.
- Does each assigned workstream appear in the output?
- Are there gaps where the roundtable identified something important but the output skips it?
- Did any agent's contribution get lost in synthesis?

**Specificity (2-3 criteria)**
Tests whether the output contains actual substance vs. consulting platitudes.
- Are recommendations tied to specific actions, not vague directives?
- Do analyses include real data (numbers, names, dates) vs. abstract assertions?
- Could someone act on this without needing a follow-up conversation?

**Evidence (1-2 criteria)**
Tests whether claims are supported.
- Are assertions backed by data, examples, or citations?
- Can the user trace a recommendation back to the analysis that supports it?

**Coherence (1-2 criteria)**
Tests whether the output holds together as a unified work product.
- Does the executive summary actually reflect what's in the body?
- Do different sections contradict each other?
- Does the narrative flow logically?

**Actionability (1-2 criteria)**
Tests whether the output is usable.
- Are next steps clear?
- Is the format appropriate for the audience?
- Does the user know exactly what to do with this?

## Grading Process

### For Each Criterion

1. **Read the criterion** — Understand exactly what it's testing
2. **Search the output** — Look for specific evidence that the criterion is met
3. **Apply the verdict**:
   - **PASS**: Clear, specific evidence exists in the output that the criterion is fully satisfied. Not just technically met — genuinely met with substance.
   - **FAIL**: Evidence is absent, contradicts the criterion, or is superficial (e.g., a recommendation section exists but contains only generic advice that could apply to any company)
4. **Cite evidence**: For PASS, quote the specific text. For FAIL, explain exactly what's missing or inadequate.
5. **Write improvement note** (FAIL only): State precisely what needs to change. Not "improve the analysis" but "add Q3 2025 revenue figures for CompetitorX and CompetitorY, and include their recent product launches from the last 6 months."

### Scorecard Format

```
GRADING — PASS [N] of 4
═══════════════════════════════════════════════
ACCURACY
  ✅ [Criterion text] — [evidence quote or summary]
  ❌ [Criterion text] — [what's missing/wrong]
     → Fix: [specific action needed]

COMPLETENESS
  ✅ [Criterion text] — [evidence]
  ✅ [Criterion text] — [evidence]

SPECIFICITY
  ❌ [Criterion text] — [what's missing/wrong]
     → Fix: [specific action needed]

EVIDENCE
  ✅ [Criterion text] — [evidence]

COHERENCE
  ✅ [Criterion text] — [evidence]

ACTIONABILITY
  ✅ [Criterion text] — [evidence]
───────────────────────────────────────────────
Score: X/Y passed | Failures: [list failed categories]
```

## Iteration Rules

### What to Fix

Only revise sections tied to failing criteria. Leave everything else untouched. Every edit must trace to a specific failing criterion.

### Regression Prevention

After revising, re-check:
1. All previously failing criteria (primary check)
2. A quick scan of previously passing criteria to ensure revisions didn't break them (regression check)

If a previously passing criterion now fails due to a revision, that's a regression — fix it before re-grading the originally failing criteria.

### Pass 4 Protocol

If the output still has failing criteria after 4 passes:

1. **Categorize the failures**:
   - **Unreasonable bar**: The criterion may be too strict for what's achievable with available information. Recommend adjusting.
   - **Data gap**: The failure reflects missing information that no amount of revision can fix (e.g., private company financials that aren't publicly available). Flag for user input.
   - **Structural issue**: The approach itself may be wrong. Recommend revisiting the roundtable.
   - **Diminishing returns**: The output is 90% there but the last 10% requires disproportionate effort. Present as-is with caveats.

2. **Present honestly**: Show the user exactly where things stand and what the options are.

3. **Recommend**: Pick the best path forward and explain why.

## Criteria Evolution

If during grading you notice:
- A criterion is trivially satisfied by mediocre output → tighten it and note to the user
- An important quality dimension has no criterion → suggest adding one
- A criterion is impossible to satisfy given the constraints → flag for the user

The criteria should get better each time you use the skill. After the loop completes, note any criteria that should be adjusted for future use.
